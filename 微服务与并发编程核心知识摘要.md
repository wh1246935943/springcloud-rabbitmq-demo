# Java 微服务与并发编程核心知识摘要

## 1. Spring Cloud 与 Lock4j

### Spring Cloud 定位
*   **本质**：是一套**微服务工具集/生态系统**，而非完整的架构蓝图。
*   **作用**：解决分布式通用技术问题（注册发现、配置、网关、熔断等）。
*   **架构设计**：业务拆分、数据一致性、组织协作需架构师自行设计。

### Lock4j 分布式锁
*   **定义**：基于 Spring Boot 的分布式锁组件，底层默认基于 **Redisson + Redis**。
*   **读法**：Lock for Java（4 谐音 for，j 代表 Java）。
*   **核心原理**：
    *   **Spring AOP**：拦截 `@Lock4j` 注解方法。
    *   **Redis 实现**：利用 Lua 脚本保证原子性，**看门狗（Watchdog）** 机制自动续期防止死锁。
*   **使用场景**：
    *   **单体单实例**：❌ 不建议。直接用 Java 原生锁（性能更高，无 Redis 依赖）。
    *   **单体多实例/微服务**：✅ 必须用。解决多 JVM 进程间的并发互斥。
*   **最佳实践**：锁粒度要细（如锁 `orderId` 而非锁整个方法）。

## 2. Spring AOP（面向切面编程）

### 核心概念
*   **定义**：一种编程范式，将**横切关注点**（日志、事务、锁）从业务逻辑中分离。
*   **原理**：基于 **动态代理**（JDK 代理或 CGLIB）。Spring 容器注入的是**代理对象**而非目标对象。
*   **切面（Aspect）**：横向切割代码，提取公共逻辑（如 Lock4j 的加锁/解锁逻辑）。

### 常见坑点与解决
*   **失效场景**：类内部方法调用（`this.method()`）会绕过代理，导致 AOP 失效。
*   **解决方案**：
    1.  **注入自己**：`@Autowired @Lazy private Service self;` 调用 `self.method()`。
    2.  **重构**：将方法拆分到另一个 Service 中（推荐）。
    3.  **AopContext**：`((Service) AopContext.currentProxy()).method()`。

## 3. Java 锁机制与原理

### 原生锁 vs 分布式锁
| 特性 | Java 原生锁 (`synchronized`/`ReentrantLock`) | 分布式锁 (Lock4j/Redis) |
| :--- | :--- | :--- |
| **适用范围** | **单 JVM 进程内** | **多 JVM 进程/集群** |
| **性能** | 纳秒级（内存/CPU 指令） | 毫秒级（网络 IO） |
| **依赖** | 无 | 依赖 Redis 中间件 |

### synchronized 原理
*   **对象头（Object Header）**：JVM 为每个对象分配的元数据（非 `java.lang.Object` 类实例）。
    *   **Mark Word**：存储锁状态（无锁/偏向/轻量/重量）、线程 ID、哈希码。
    *   **Klass Pointer**：指向类元数据。
*   **锁升级**：偏向锁 → 轻量级锁（CAS 自旋）→ 重量级锁（操作系统 Mutex 阻塞）。
*   **锁竞争**：多线程争夺同一把锁。激烈竞争会导致上下文切换，性能雪崩。

### ReentrantLock 原理
*   **核心**：基于 **AQS**（AbstractQueuedSynchronizer）。
*   **实现**：`volatile state` + **CAS** + **CLH 双向队列**。
*   **优势**：可中断、公平锁、多条件变量。

## 4. 无锁编程与并发容器

### CAS 与 AtomicInteger
*   **CAS (Compare-And-Swap)**：CPU 原子指令，“认为值是 A 则改成 B"，失败则重试。
*   **AtomicInteger**：
    *   **本质**：线程安全的整数**类**（引用类型），内部维护 `volatile int`。
    *   **原理**：基于 CAS 自旋，**无锁**，失败不阻塞。
    *   **适用**：简单计数、状态标志。竞争激烈时消耗 CPU。
    *   **对比**：`int`（不安全） < `Integer`（不可变，不安全） < `AtomicInteger`（安全，可变）。

### ConcurrentHashMap (CHM)
*   **HashMap 不安全**：多线程 `put` 导致数据覆盖（所有版本），JDK1.7 可能死循环。
*   **CHM 安全原理**：
    *   **JDK 1.7**：分段锁（Segment），锁粒度较粗。
    *   **JDK 1.8**：**数组 + 链表 + 红黑树**。锁粒度细化到**桶（Node）**。
    *   **实现**：空桶用 **CAS** 插入，非空桶用 **`synchronized`** 锁住头节点。
*   **特性**：不允许 key/value 为 null（避免并发歧义）。
*   **演进路线**：粗锁 (`synchronized`) → 细锁 (`CHM`) → 无锁 (`Atomic`)。

## 5. 核心结论
1.  **架构**：Spring Cloud 是工具，架构需自行设计。
2.  **锁选型**：单机用原生锁，集群用分布式锁。
3.  **AOP**：注意内部调用失效问题，理解代理机制。
4.  **并发优化**：优先无锁（CAS），其次细锁（CHM），最后粗锁。减少锁竞争是性能关键。
